# Foundry .t.sol importer — implementation plan

## Goal

Let users paste a Foundry test file (`.t.sol`), run it against the in-browser
EVM, and see pass/fail output — no install, no Foundry binary.

## Why this is non-trivial

Foundry tests depend on two cooperating things:

1. **`Test` base contract** (from forge-std) — provides `assertEq`, `assertGt`,
   `assertTrue`, `expectRevert`, `expectEmit`, and ~80 other assertions.
2. **`vm` cheatcodes** — an opaque address (`0x7109...`) the test contract
   calls; Foundry intercepts these calls and performs host-side state changes
   (warp time, set balance, impersonate caller, mock returns, etc.).

To run `.t.sol` in the browser, we need to emulate both.

## MVP scope (what we implement)

A **subset** that covers 80% of test files:

### Assertions (ship in forge-std shim)
```
assertTrue(bool)
assertFalse(bool)
assertEq(uint,uint) / assertEq(int,int) / assertEq(address,address)
assertEq(bytes32,bytes32) / assertEq(string,string) / assertEq(bytes,bytes)
assertGt / assertGe / assertLt / assertLe
assertNotEq
fail(string)
```

Each is a pure-Solidity function emitting a `log_named_*` event when it
fails. The runner watches the event log for failures.

### Cheatcodes (intercept at EVM.call)
```
vm.prank(address)             — set msg.sender for next call
vm.startPrank / vm.stopPrank  — persistent prank
vm.deal(address, uint256)     — set balance
vm.warp(uint256)              — set block.timestamp
vm.roll(uint256)              — set block.number
vm.expectRevert()             — expect next call to revert
vm.expectRevert(bytes)        — expect specific revert reason
vm.mockCall(addr, data, retData) — return `retData` on matching call
vm.assume(bool)               — skip fuzz iteration if false
```

### NOT in MVP
- `vm.ffi` (shell execution)
- `vm.readFile` / `vm.writeFile`
- `vm.envX` (env vars)
- `vm.createSelectFork` (fork mode — separate project)
- `vm.broadcast` (deployment scripts)
- `invariant_*` tests — we have a separate Invariants tool
- Fuzz-specific cheatcodes (`vm.assume` is included, others deferred)

## Architecture

```
┌──────────────────────────────────────────────────────────────┐
│ User pastes Contract.sol + Test.t.sol into the tool          │
├──────────────────────────────────────────────────────────────┤
│ 1. Inject forge-std shim (assertEq suite) as a prefix source │
│ 2. Compile all three via existing solc worker                │
│ 3. Deploy the test contract via LocalEVM.deploy()            │
│ 4. Identify test_* functions from the ABI                    │
│ 5. For each: call setUp() if present, then the test fn       │
│    – override LocalEVM to intercept calls to VM_ADDRESS       │
│    – track expectRevert / prank / warp state                  │
│ 6. Parse emitted log_* events to collect assertion failures   │
│ 7. Render pass/fail grid with per-test logs                  │
└──────────────────────────────────────────────────────────────┘
```

### Cheatcode interception

Extend `LocalEVM` with a hook:

```js
class LocalEVM {
  setCheatcodeHandler(fn) { this._cheatHandler = fn }
  // In .call()/.traceCall(), intercept when to == VM_ADDRESS
  // before passing to evm.runCall — decode selector, dispatch to handler.
}
```

The handler maintains state: `{ pranked, prankDuration, expectRevert,
mockCalls, warp, roll }`. Next `runCall` consults this state to rewrite
caller / timestamp / block number.

**Tricky bit**: `vm.expectRevert()` must make the next call that reverts
count as a *pass*. Implementation: set a flag; wrap the next user-initiated
call, invert the success check.

### forge-std assertion shim

```solidity
// forge-std-shim.sol (generated by the tool, prepended to sources)
abstract contract Test {
    event log_named_uint(string key, uint value);
    event log_string(string msg);
    // ...
    function assertEq(uint a, uint b) internal {
        if (a != b) {
            emit log_string("assertEq failed");
            emit log_named_uint("expected", b);
            emit log_named_uint("actual", a);
            fail();
        }
    }
    // ...
}
```

When a `log_string("assertion failed")` event appears in the test's log
output, the runner marks that test as failed.

## File structure

```
js/tools/foundry-test.js        — the tool UI + runner
js/shared/forge-std-shim.sol    — bundled assertion library (as JS string)
js/shared/local-evm.js          — extended with setCheatcodeHandler()
js/shared/cheatcodes.js         — vm.* dispatch + state
```

## Complexity estimate

- forge-std shim (Solidity source): **~400 LOC** (mostly assertion boilerplate)
- cheatcodes.js dispatcher: **~300 LOC**
- LocalEVM hook + call interception: **~150 LOC** of changes
- foundry-test.js UI + runner: **~400 LOC**
- Test fixtures + docs: **~200 LOC**

**Total: ~1500 LOC for the MVP.**

## Risk / uncertainty

1. **Cheatcode compatibility**: real Foundry keeps adding cheatcodes. The
   MVP covers the common ones, but users will paste tests that use newer
   ones and hit "not implemented" errors. Needs a clear error message listing
   what's supported.
2. **expectEmit accuracy**: matching emitted events with checked topics is
   fiddly; would defer to v2.
3. **`setUp()` state isolation**: Foundry runs setUp before every test,
   we'd need to snapshot/revert between tests (LocalEVM already has this).
4. **Stack traces**: Foundry shows line numbers when assertions fail. Ours
   can reference log_* event line numbers but not call-site trace. Would
   need source-map integration (already built in the disassembler) to
   synthesize.

## Suggested rollout

- **v1**: assertions + 6 most-used cheatcodes (`prank`, `deal`, `warp`,
  `expectRevert`, `startPrank`, `stopPrank`). Ship and gather feedback.
- **v2**: `mockCall`, `expectEmit`, better error UX, source-linked traces.
- **v3**: fuzzing, `vm.assume`, random-input generators.

## Why we're deferring

The assertion shim alone is a week of solid work if we want to match
forge-std's full API (and tests break when the API doesn't match exactly).
It's a larger project than can be completed in a single session alongside
other work, and a partial implementation would mislead users into thinking
their tests are passing when they silently skip assertions.
